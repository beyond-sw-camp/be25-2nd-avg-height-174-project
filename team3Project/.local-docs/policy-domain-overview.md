# policy-domain-overview.md

## 문서 목적
이 문서는 정책(Policy) 도메인의 역할과 범위, 핵심 데이터, API 방향, 다른 도메인과의 연결을 빠르게 파악하기 위한 개인용 개요 문서입니다. 현재 이 프로젝트에서 가장 구체적으로 정리되고 있는 영역 중 하나이므로, 이후 구현과 문서 갱신의 기준점으로 사용합니다.

## 정책 도메인이 필요한 이유
구매대행 시스템에서는 상품마다 반복적으로 계산하고 판단해야 하는 기준이 많습니다. 예를 들면 판매가 계산, 마진 확보, 금지어 제거, 치환어 적용, 배송비 설정, 손실 방지 규칙 등이 있습니다.

정책 도메인은 이러한 기준값을 사용자 단위로 관리하고, 상품 가공과 등록, 주문 처리 흐름에서 재사용할 수 있게 만드는 역할을 합니다.

즉, 정책 도메인은 단순 설정 화면이 아니라 전체 자동화 흐름의 기준값 제공자입니다.

## 정책 도메인이 영향을 주는 흐름
- 상품 소싱 이후 가공 단계
- 판매가 계산 단계
- 마켓 등록 단계
- 손실 방지 / 자동 보호 판단
- 수익 분석 및 실제 마진 확인

## 핵심 정책 항목
현재 기준으로 정책 도메인에서 중요하게 다뤄야 하는 항목은 다음과 같습니다.

### 1. 마진 관련 정책
- 목표 마진율
- 최소 마진 금액
- 마켓 수수료율
- 카드 수수료율

### 2. 가격 계산 보조 정책
- 환율
- 단위 올림 단위

### 3. 자동 보호 설정
- 최소 마진 보호
- 환율 변동 시 가격 자동 업데이트
- 손실 발생 시 판매 중지

### 4. 배송비 정책
- 배송비 종류
- 기본 배송비
- 제주 및 도서산간 추가 배송비
- 반품비

### 5. 텍스트 가공 정책
- 금지어
- 치환어

### 6. AI 기반 자동 최적화 설정
- Amazon 가격 기준 자동 조정
- 경쟁 상품 가격 자동 조정

## 금지어 / 치환어의 역할
금지어와 치환어는 상품명, 옵션명, 상세설명 등 텍스트 가공 과정에서 사용될 수 있습니다.

### 금지어
- 등록이 제한되거나 노출하면 안 되는 단어를 걸러내기 위한 기준
- 현재 기준으로 상품명 포함 시 수집, 가공, 업로드, 결제 과정에서 제외 처리 가능

### 치환어
- 특정 단어를 더 적절한 표현으로 바꾸기 위한 기준
- 상품명 정제, 마켓 정책 대응, 표현 통일 등에 사용 가능

## 가격 계산과의 연결
정책 도메인은 판매가 계산 로직과 직접 연결됩니다. 현재까지의 요구사항 기준으로, 가격 계산에는 대체로 아래 요소가 들어갑니다.

- 원가
- 환율
- 목표 마진율
- 최소 마진 금액
- 마켓 수수료율
- 카드 수수료율
- 단위 올림 단위
- 배송비 정책
- AI 기반 자동 최적화 설정 중 Amazon 가격 기준 자동 조정

중요한 점은, 정책 도메인은 이 값을 저장하는 역할과 계산에 필요한 기준을 제공하는 역할을 맡고, 실제 계산 구현은 별도 계산 로직 또는 서비스에서 수행될 가능성이 높다는 점입니다.

## 자동 보호 설정과의 연결
자동 보호 설정은 단순 체크박스 기능이 아니라 주문/등록 흐름을 제어할 수 있는 정책입니다.

예상 연결 지점은 다음과 같습니다.
- 마진이 최소 기준보다 낮아지면 경고 또는 차단
- 환율 기준이 바뀌면 판매가 재계산 또는 업데이트
- 손실 가능성이 생기면 판매 중지 또는 자동 주문 차단

현재 기준으로 환율 변동 시 가격 자동 업데이트는 더미 등록 DB의 판매가 갱신까지 포함합니다.

## 배송비 정책과의 연결
배송비 정책은 판매가와 별도 영역이지만, 실제 사용자 입장에서는 판매 전략과 수익 계산에 함께 영향을 줍니다.

따라서 정책 도메인에서는 배송비 관련 기준을 별도 정책으로 다루되, 상품 등록과 수익 분석에 연결될 수 있도록 설계하는 것이 중요합니다.

현재 1차 구현 기준 배송비 유형은 무료배송 / 유료배송으로 둡니다.
조건부 무료배송 기준은 구조 검토 대상이지만, 이번 단계 실제 구현에서는 제외합니다.

## 정책 도메인의 데이터 관점
정책 도메인은 크게 다음 단위로 나눠볼 수 있습니다.

### 1. 마켓별 설정값
- 사용자 + 마켓 코드 단위 판매 정책
- 사용자 + 마켓 코드 단위 배송비 정책
- 사용자 + 마켓 코드 단위 자동 보호 설정
- 사용자 + 마켓 코드 단위 AI 기반 자동 최적화 설정

### 2. 목록성 설정값
- 사용자별 금지어 목록
- 사용자별 치환어 목록

### 3. 묶음 조회 관점
실제 화면이나 가공 로직에서는 여러 정책을 한 번에 필요로 할 수 있으므로, 단건 조회뿐 아니라 묶음 조회(Bundle) 관점도 중요합니다.

현재 정책 설정 화면은 먼저 국내 마켓을 선택하고, 그 뒤에 선택한 마켓의 정책을 입력하는 구조를 기준으로 봅니다.

1차 국내 마켓 목록은 다음과 같습니다.
- NAVER_SMART_STORE
- COUPANG
- ELEVEN_STREET
- GMARKET
- AUCTION

화면 기본 선택 마켓과 현재 기본 테스트 마켓은 COUPANG으로 둡니다.

## 정책 API 범위
현재 기준으로 필요한 API 범위는 대체로 다음과 같습니다.

### 기본 정책 설정 API
- 마켓 목록 조회
- 특정 마켓 정책 조회
- 특정 마켓 정책 저장 또는 수정

### 금지어 API
- 금지어 등록
- 금지어 목록 조회
- 금지어 수정
- 금지어 삭제

### 치환어 API
- 치환어 등록
- 치환어 목록 조회
- 치환어 수정
- 치환어 삭제

### 정책 적용 / 가공 API
- 선택한 마켓의 정책 묶음 조회
- 상품 가공 요청 시 마켓별 가격 정책 + 사용자 공통 텍스트 정책 동시 적용
- 가공 결과를 더미 등록 DB에 저장
- 현재 가공 요청 DTO는 번역 완료된 상품명/브랜드, 원가, 통화, 이미지 정보를 받는 방향으로 정리한다.
- 현재 가공 응답 DTO는 제외 여부, 제외 사유, 가공 결과 텍스트, 원가 통화, 적용 환율, 환율 적용 원가, 판매가, 배송비, 등록 상태를 반환한다.

## 다른 도메인과의 연결
### 상품 가공 도메인
가장 직접적으로 연결되는 영역입니다.
- 금지어 적용
- 치환어 적용
- 판매가 계산 기준 제공
- 배송비 정책 반영
- 자동 보호 설정 반영
- AI 기반 자동 최적화 설정 반영

### 마켓 등록 도메인
- 최종 판매가 반영
- 배송비 정책 반영
- 등록 가능 여부 판단
- 이번 프로젝트에서는 실제 외부 마켓 API 등록이 아니라 더미 등록 DB 저장 방식으로 연결

### 주문 / 자동결제 도메인
- 손실 방지 정책 반영
- 자동 보호 규칙 반영

### 수익 분석 도메인
- 목표 마진 대비 실제 결과 비교
- 월별 수익과 실제 마진 계산 기준 제공

## 현재 구현 시 유의할 점
1. 정책 저장과 실제 적용 로직을 분리해서 보기
2. 사용자 + 마켓 코드 단위 정책과 사용자 공통 텍스트 정책을 구분하기
3. 금지어 / 치환어는 단순 CRUD를 넘어 가공 흐름 연결까지 고려하기
4. 구현 제외 기능도 화면, 요청 구조, 저장 구조 어디까지 둘지 구분해서 설계하기
5. 설정 화면 요구사항과 실제 API 구조가 어긋나지 않게 하기
6. 이후 Redis를 쓴다면 캐시 대상이 무엇인지 먼저 분명히 하기
7. 현재 단계에서는 정책 수정 직후 정확성을 우선하여 캐시 없이 구현하고, 이후 정책 묶음 조회 부하가 커지면 Redis 기반 공유 캐시를 검토하기

현재 기준으로 구현 범위는 다음과 같이 봅니다.
- 실제 구현: 마켓별 정책 저장/조회, 금지어/치환어 CRUD, 상품 가공 시 금지어/치환어/환율/마진/수수료/단위 올림 반영, 최소 마진 보호, 금지어 차단 시 조기 반환, 상품 가공 서비스 테스트
- 저장만 우선: 손실 발생 시 실제 외부 마켓 판매중지 연동
- 화면/구조 우선: 더미 등록 DB 저장, 경쟁 상품 가격 자동 조정, 마켓별 실제 수수료 자동 조회 연동, Eureka 기반 소싱 서비스 조회 연동

## 현재 상태 정리
### 확정에 가까운 내용
- 정책 도메인이 핵심 도메인이라는 점
- 금지어 / 치환어 / 마진 / 환율 / 단위 올림 / 배송비 / 자동 보호 설정 / AI 기반 자동 최적화 설정이 필요하다는 점
- 사용자별 정책 관리가 필요하다는 점
- 일부 정책은 사용자 공통이 아니라 사용자 + 마켓 코드 단위라는 점
- 상품 가공 이후 실제 외부 마켓 등록 대신 더미 등록 DB 저장으로 현재 범위를 제한한다는 점

### 진행 중인 내용
- API 구조 정리
- 정책 데이터 묶음 조회 방식
- 다른 도메인과의 연결 방식
- 더미 기준값을 활용한 Amazon 가격 자동 조정 방식
- 가공 서비스와 소싱 서비스 간 Eureka 기반 연동 계약
- 가공 결과와 더미 등록 DB 저장 구조 연결

### 검토 필요
- 정책 세부 엔티티 분리 범위
- 자동 보호 로직의 정확한 조건
- 정책 캐시 전략
- 현재는 캐시 없이 구현하고, 이후 Redis 기반 공유 캐시 도입 시점 검토
- MSA 분리 시 정책 서비스 경계
- 조건부 무료배송 기준의 도입 시점과 실제 계산 규칙
- 더미 등록 DB에 저장할 계산 핵심 값 범위
- 번역/정규화 완료 소싱 데이터를 어떤 서비스 조회 방식으로 받을지
- 가공 서비스가 소싱 서비스 DTO를 직접 의존할지, 별도 내부 DTO로 한 번 변환할지
