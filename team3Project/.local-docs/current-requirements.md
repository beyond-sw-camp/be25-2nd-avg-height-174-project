# current-requirements.md

## 문서 목적
이 문서는 현재 시점에서 합의되었거나 강하게 요구되고 있는 기능 요구사항을 정리하는 개인용 기준 문서입니다. 모든 내용이 최종 확정은 아니며, 확정된 내용과 검토가 필요한 내용을 구분해서 관리합니다.

## 서비스 목표
해외 구매대행 업무를 하나의 흐름으로 연결하여, 사용자가 상품을 소싱하고 가공하여 등록용 데이터로 만든 뒤 주문을 수집하고 발주/결제/배송 추적/수익 관리까지 이어갈 수 있도록 지원하는 시스템을 만든다.

## 해결하려는 사용자 문제
- 여러 소싱 사이트와 마켓을 오가며 수작업으로 처리해야 하는 번거로움
- 판매가 계산과 마진 관리의 복잡성
- 금지어/치환어 관리의 반복 작업
- 주문, 발주, 실패 내역 추적의 비효율
- 자동화 흐름이 끊겨 운영 피로도가 높아지는 문제

## 핵심 사용자
- 해외 구매대행 운영자
- 반복적인 상품 등록과 주문 처리를 줄이고 싶은 셀러
- 마진 / 금지어 / 배송비 / 환율 정책을 직접 관리하고 싶은 사용자

## 전체 기능 범위
- 사용자 인증 및 계정 관리
- 정책 설정 관리
- 카테고리 / 상품 소싱
- 상품 가공
- 마켓 등록
- 전체 주문 관리
- 자동 주문 / 자동 결제
- 배송 추적
- 실패 내역 관리
- 수익 / 마진 분석

## 요구사항 정리

### 1. 사용자 / 인증
#### 확정
- 사용자 식별이 가능해야 함
- 사용자별 정책 데이터가 분리되어야 함

#### 검토 필요
- 로그인 방식의 최종 구조
- 인증/인가 범위와 토큰 처리 방식

### 2. 정책 설정
#### 확정
사용자는 자신의 판매 정책을 설정할 수 있어야 한다.

정책 설정 구조는 다음 기준을 따른다.
- 사용자는 먼저 국내 마켓을 선택한 뒤 해당 마켓의 정책을 설정한다.
- 마켓별 정책은 사용자 + 마켓 코드 단위로 저장한다.
- 금지어 / 치환어는 사용자 공통 정책으로 관리한다.
- 국내 마켓 목록은 코드 고정 방식으로 관리한다.

1차 국내 마켓 목록은 다음과 같다.
- NAVER_SMART_STORE
- COUPANG
- ELEVEN_STREET
- GMARKET
- AUCTION

화면 기본 선택 마켓과 현재 구현 우선순위 기준 기본 테스트 마켓은 COUPANG으로 둔다.

마켓별 정책에는 다음 항목이 포함된다.
- 목표 마진율
- 최소 마진 금액
- 마켓 수수료율
- 카드 수수료율
- 환율
- 단위 올림 단위
- 자동 보호 설정
- 배송비 정책
- AI 기반 자동 최적화 설정

사용자 공통 정책에는 다음 항목이 포함된다.
- 금지어
- 치환어

마켓별 정책 항목의 현재 기준은 다음과 같다.
- 목표 마진율: 정수 입력, 기본값 30, 1 이상 100 이하
- 최소 마진 금액: 정수 입력, 기본값 5000, 음수 불가
- 마켓 수수료율: 숫자 입력, 기본값 10
- 카드 수수료율: 실수 입력, 기본값 3.3
- 환율: 기본값 1350, 선택 지정 + 직접 입력 가능
- 단위 올림 단위: 만원 / 천원 / 100원 / 50원 / 10원 중 선택
- 자동 보호 설정: 최소 마진 보호 / 환율 변동 시 가격 자동 업데이트 / 손실 발생 시 판매 중지
- 배송비 정책: 무료배송 / 유료배송, 기본 배송비, 제주 및 도서산간 추가 배송비, 반품비
- AI 기반 자동 최적화 설정: Amazon 가격 기준 자동 조정, 경쟁 상품 가격 자동 조정

현재 구현 기준 배송비 유형은 무료배송 / 유료배송만 우선 구현한다.
조건부 무료배송 기준은 이번 단계에서는 구현 대상에서 제외하고, 추후 시간이 나면 확장할 항목으로 둔다.

AI 기반 자동 최적화 설정의 현재 기준은 다음과 같다.
- Amazon 가격 기준 자동 조정: 더미 값 기준으로라도 실제 계산 로직을 구현한다.
- 경쟁 상품 가격 자동 조정: 화면 및 구조는 두되 실제 연산 로직은 구현하지 않는다.

#### 검토 필요
- 정책 조회를 묶음(Bundle)으로 제공할지 여부
- 마켓별 기본 수수료율 자동 주입 방식
- 조건부 무료배송 기준의 도입 시점과 실제 적용 규칙

### 3. 금지어 / 치환어
#### 확정
- 사용자는 금지어를 등록, 조회, 수정, 삭제할 수 있어야 한다.
- 사용자는 치환어를 등록, 조회, 수정, 삭제할 수 있어야 한다.
- 금지어 / 치환어는 상품 가공 단계에서 활용될 수 있어야 한다.
- 금지어는 사용자별 최대 2000개 수준까지 관리할 수 있어야 한다.
- 치환어는 사용자별 최대 1000개 수준까지 관리할 수 있어야 한다.
- 금지어는 현재 기준으로 상품명에 우선 적용한다.
- 치환어는 원본 단어 + 치환 단어 쌍으로 관리하고, 현재 기준으로 상품명에 우선 적용한다.
- 금지어 항목당 최대 길이는 50자 제안을 기준으로 본다.
- 치환어 항목당 최대 길이는 100자 제안을 기준으로 본다.

#### 검토 필요
- 시스템 공통 금지어와 사용자별 금지어를 분리할지 여부
- 적용 우선순위와 예외 규칙
- 치환어 저장 시 금지어와 충돌하는 단어를 어떻게 처리할지

### 4. 가격 계산
#### 확정
판매가 계산에는 다음 요소가 반영되어야 한다.
- 원가
- 환율
- 목표 마진율
- 최소 마진 금액
- 마켓 수수료율
- 카드 수수료율
- 단위 올림 단위
- 배송비 정책
- AI 기반 자동 최적화 설정 중 Amazon 가격 기준 자동 조정

#### 검토 필요
- 계산식의 정확한 공식
- 소수점 처리 방식
- 환율 업데이트 방식
- Amazon 가격 기준 자동 조정 시 사용할 더미 기준값 구조

### 5. 자동 보호 설정
#### 확정
자동 보호 설정에는 다음 성격의 기능이 필요하다.
- 최소 마진 보호
- 환율 변동 시 가격 자동 업데이트
- 손실 발생 시 판매 중지

자동 보호 설정의 현재 구현 방향은 다음과 같다.
- 최소 마진 보호: 실제 계산과 가공 흐름에서 반영한다.
- 환율 변동 시 가격 자동 업데이트: 환율 값이 바뀌면 재계산을 수행하고 더미 등록 DB의 판매가까지 갱신한다.
- 손실 발생 시 판매 중지: 외부 마켓 연동은 하지 않더라도 가공 / 더미 등록 흐름 차단과 상태값 반영까지 구현한다.

#### 검토 필요
- 자동 업데이트 시점
- 손실 판단 기준
- 판매 중지 후 복구 조건

### 6. 배송비 정책
#### 확정
배송비는 판매가와 별도로 관리되는 정책 영역이다.

포함 항목은 다음을 기준으로 한다.
- 배송비 종류
- 기본 배송비
- 제주 및 도서산간 추가 배송비
- 반품비

현재 1차 구현 기준 배송비 종류는 무료배송 / 유료배송으로 둔다.

#### 검토 필요
- 마켓별 배송비 정책 차이 반영 여부
- 판매가와 배송비의 결합 처리 방식

### 7. 상품 소싱
#### 확정
- 사용자는 카테고리 또는 상품 기준으로 소싱할 수 있어야 한다.
- 소싱 사이트 연동 상태를 확인할 수 있어야 한다.

#### 검토 필요
- 지원할 소싱 사이트 범위
- 원본 데이터 저장 방식
- 수집 주기와 동기화 방식

### 8. 상품 가공
#### 확정
- 가공 단계에서 정책이 적용되어야 한다.
- 금지어 / 치환어 / 가격 정책 / 배송비 정책이 영향을 줄 수 있어야 한다.
- 가공 결과는 실제 국내 마켓 등록 대신 더미 등록 DB에 저장할 수 있어야 한다.
- 더미 등록 DB에는 정책 ID만이 아니라 계산 당시 핵심 값도 함께 저장할 수 있어야 한다.
- 현재 가공 요청은 소싱 서비스에서 번역/정규화가 끝난 결과를 입력으로 받는 방향으로 정리한다.
- 현재 가공 응답은 제외 여부, 제외 사유, 가공된 상품명/브랜드, 원가 통화, 적용 환율, 환율 적용 원가, 판매가, 배송비, 등록 상태를 포함한다.
- 현재 판매가 계산은 배송비를 판매가와 분리하고, 환율, 목표 마진율, 최소 마진 보호 여부, 마켓/카드 수수료율, 단위 올림을 반영한다.
- 금지어 포함 시 상품 가공은 조기 차단하고 등록 상태를 `BLOCKED`로 반환한다.

더미 등록 DB에 함께 저장할 핵심 값 예시는 다음과 같다.
- 적용 마켓
- 계산 당시 환율
- 목표 마진율
- 마켓 수수료율
- 카드 수수료율
- 배송비
- 계산된 판매가
- 자동 조정 적용 여부
- 등록 상태

#### 검토 필요
- 자동 가공과 수동 수정의 경계
- 원본 데이터와 가공 데이터의 분리 방식
- 더미 등록 DB 저장 시 가공 응답 구조와 저장 구조를 어디까지 동일하게 맞출지
- 번역 완료된 소싱 결과를 Eureka 기반으로 어떤 조회 API/DTO 계약으로 받을지

### 9. 마켓 등록
#### 확정
- 가공된 상품을 등록용 데이터로 만들 수 있어야 한다.
- 등록 상품 관리 화면이 필요하다.
- 이번 프로젝트에서는 실제 외부 마켓 API 등록 대신 더미 등록 DB 저장 방식으로 진행한다.

#### 검토 필요
- 등록 실패 재처리 흐름
- 마켓별 필수 입력값 차이

### 10. 주문 / 발주 / 자동결제
#### 확정
- 전체 주문 페이지가 필요하다.
- 자동 주문 현황을 확인할 수 있어야 한다.
- 결제 수단 관리와 자동 결제 설정이 필요하다.
- 실패 내역을 확인할 수 있어야 한다.

#### 검토 필요
- SKU 매칭 정확도 관리
- 실패 상태 코드 설계
- 자동 결제 승인/차단 기준

### 11. 배송 추적
#### 확정
- 주문 이후 배송 상태 추적이 필요하다.

#### 검토 필요
- 추적 대상 단계 정의
- 사용자 화면에서 보여줄 상태값 설계

### 12. 수익 / 마진 관리
#### 확정
- 실제 마진 확인 기능이 필요하다.
- 월별 수익 확인 기능이 필요하다.

#### 검토 필요
- 실현 수익 기준 시점
- 환불 / 취소 반영 기준

## 비기능 요구사항
### 확정 또는 강한 요구
- 사용자별 데이터 분리
- 중복 등록 방지
- 예외 상황 식별 가능성
- 팀 개발에 맞는 구조적 분리

### 검토 필요
- 성능 최적화 방식
- Redis 사용 지점
- 현재 단계에서는 정책/상품 가공 흐름의 정확성을 우선하여 캐시 없이 구현하고, 반복 조회 부하가 확인되면 Redis 기반 공유 캐시 도입을 검토한다.
- 캐시 만료 전략
- 로컬 캐시는 다중 인스턴스 환경 일관성 문제 때문에 장기 기본안으로 두지 않는다.
- MSA 분리 기준

## 현재 남아 있는 큰 작업
1. 마켓별 정책 엔티티 / DTO / API 구조 확정
2. 상품 가공 결과를 더미 등록 DB에 저장하는 구조와 상태값 설계
3. Eureka 기반으로 소싱 서비스의 번역/정규화 결과를 조회하는 연동 계약 확정
